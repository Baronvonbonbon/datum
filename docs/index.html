DATUM

Decentralized advertising protocol on Polkadot Hub. On-chain settlement, privacy-preserving engagement, no intermediaries.

Mission

The web already works. People publish content, others read it, and advertising connects the two. Search engines, social networks, news sites, forums — these experiences are familiar and valuable. What isn't necessary is the centralized intermediary extracting rent from every interaction.

DATUM explores how these familiar web experiences — publishing, browsing, advertising, content discovery — can continue to function as they always have, but with on-chain settlement replacing opaque middlemen. Publishers keep their sites. Advertisers keep their campaigns. Users keep their data. The protocol handles the settlement, governance, and trust — without a dedicated service provider sitting in the middle.

No platform lock-in. No data extraction. No intermediary taking a cut for matching two parties that could match themselves. Just the existing web, with transparent on-chain infrastructure underneath.

Walkthrough

How It Works

1
Publish
Add the SDK tag to your site. Declare your content tags and publisher address. Your site serves ads directly — no ad server needed.
2
Campaign
Advertisers deposit DOT into an on-chain escrow, set a CPM bid and required tags. Community governance votes to activate or reject campaigns.
3
Engage
Users browse normally with the DATUM extension. Impressions are tracked locally with engagement scoring. Behavior data never leaves the browser.
4
Settle
Claim hash chains are submitted on-chain. Revenue splits automatically: publisher, user, and protocol — all transparent, all verifiable.

Protocol Architecture

Seventeen smart contracts deployed on Polkadot Hub (PolkaVM / pallet-revive):

  • PauseRegistry — Emergency circuit breaker
  • Timelock — Governance-delayed parameter changes
  • Publishers — Publisher registration, take rates, blocklist/allowlists
  • Campaigns — Campaign lifecycle, metadata, status management
  • BudgetLedger — Campaign escrow, daily caps, settlement tracking
  • PaymentVault — Pull-payment vault (publisher/user/protocol balances)
  • CampaignLifecycle — Complete/terminate/expire, inactivity timeout
  • AttestationVerifier — Mandatory publisher co-signature for all settlements
  • GovernanceV2 — Conviction-weighted voting (9 levels), anti-grief termination protection
  • GovernanceSlash — Symmetric slash on losing voters
  • Settlement — Blake2-256 hash-chain validation, three-way revenue split
  • Relay — EIP-712 gasless settlement with publisher co-signature
  • ZKVerifier — Engagement proof verification (stub, Groth16 post-alpha)
  • TargetingRegistry — Tag-based publisher/campaign targeting (AND-logic matching)
  • CampaignValidator — Campaign creation validation satellite
  • ClaimValidator — Claim validation satellite (Blake2-256 on PolkaVM)
  • GovernanceHelper — Slash computation + dust guard satellite

Publisher Integration

Add the SDK to any page with two lines:

<script src="https://your-cdn/datum-sdk.js"
  data-tags="crypto-web3,defi,en"
  data-publisher="0xYOUR_PUBLISHER_ADDRESS"
  data-relay="https://your-relay.example.com"></script>
<div id="datum-ad-slot"></div>

data-tags declares which tags describe your site (comma-separated values from the tag dictionary below). data-publisher is your registered on-chain address. data-relay is the URL of your publisher relay endpoint (the extension uses this to request attestations and submit claims — no extension rebuild needed when the URL changes). The extension matches campaigns by tag overlap (AND-logic), runs a Vickrey auction, and injects the winning ad into the slot.

Tag Dictionary

Publishers and campaigns declare targeting using tags from four dimensions. Tags are keccak256("dimension:value") hashes stored on-chain via the TargetingRegistry contract. Publishers can set up to 32 tags; campaigns can require up to 8. A campaign matches a publisher when the publisher has all of the campaign's required tags (AND-logic).

Publisher Relay

Publishers operate a publicly available relay endpoint that co-signs user claim batches (EIP-712 attestation) and submits them on-chain via DatumRelay.settleClaimsFor(). The relay pays gas so users don't need to — the publisher is reimbursed from the campaign budget's take-rate share. How a publisher chooses to run their relay — automated service, manual dashboard, serverless function, or anything else — is entirely up to them.

How It Works

  • User browses a publisher page with the DATUM SDK and extension installed
  • Impressions are tracked locally; claims accumulate in the extension
  • When the user signs a claim batch (EIP-712), the extension requests a publisher co-signature from the relay's /.well-known/datum-attest endpoint
  • The signed batch is sent to /relay/submit, where the relay verifies the user's EIP-712 signature and processes it for on-chain submission
  • Settlement splits revenue: publisher (take rate), user (75% of remainder), protocol (25%)

Security

  • No private key exposure: Users sign claim batches locally using EIP-712 typed data. Only the signature is sent to the relay — private keys never leave the browser
  • Signature verification: The relay verifies every user signature before queuing. Forged or tampered batches are rejected
  • Rate limiting: Per-IP sliding-window limits (10 attestation/min, 5 submit/min)
  • Input validation: Address format, nonce range, batch size caps (max 100 claims per batch)
  • HTTPS only: The extension rejects HTTP relay URLs for non-local domains

Relay Downtime

If the publisher relay is offline (check the indicator above), claims will still accumulate in the extension — they can be submitted when the relay comes back online, or the user can submit claims directly on-chain. No claims or earnings are lost due to relay downtime.

Run Your Own Relay

A reference relay implementation is included in the repository at docs/relay-bot-template/. This is one way to run a publisher relay — an automated Node.js service that queues and submits batches on a timer. Publishers are free to implement their relay endpoint however they prefer (serverless, manual review, custom tooling, etc.) as long as it exposes the required HTTP endpoints. To set up the reference implementation: 

# 1. Register as a publisher on-chain (via the DATUM extension Publisher tab)
# 2. Copy the template
cp -r docs/relay-bot-template my-relay
cd my-relay

# 3. Install dependencies
npm install

# 4. Configure your publisher key and contract addresses
#    Edit .env (see .env.example in the template)
export PUBLISHER_KEY="0xYOUR_PRIVATE_KEY"
export RPC_URL="https://eth-rpc-testnet.polkadot.io/"

# 5. Start the relay
node relay-bot.mjs

# 6. (Optional) Expose publicly via reverse proxy, tunnel, or cloud hosting

See the template's README.md for full configuration options, systemd setup, and security hardening details.
