DATUM

Decentralized advertising protocol on Polkadot Hub. On-chain settlement, privacy-preserving engagement, no intermediaries.

Mission

The web already works. People publish content, others read it, and advertising connects the two. Search engines, social networks, news sites, forums — these experiences are familiar and valuable. What isn't necessary is the centralized intermediary extracting rent from every interaction.

DATUM explores how these familiar web experiences — publishing, browsing, advertising, content discovery — can continue to function as they always have, but with on-chain settlement replacing opaque middlemen. Publishers keep their sites. Advertisers keep their campaigns. Users keep their data. The protocol handles the settlement, governance, and trust — without a dedicated service provider sitting in the middle.

No platform lock-in. No data extraction. No intermediary taking a cut for matching two parties that could match themselves. Just the existing web, with transparent on-chain infrastructure underneath.

Walkthrough

How It Works

1
Publish
Add the SDK tag to your site. Declare your content categories and publisher address. Your site serves ads directly — no ad server needed.
2
Campaign
Advertisers deposit DOT into an on-chain escrow, set a CPM bid and category. Community governance votes to activate or reject campaigns.
3
Engage
Users browse normally with the DATUM extension. Impressions are tracked locally with engagement scoring. Behavior data never leaves the browser.
4
Settle
Claim hash chains are submitted on-chain. Revenue splits automatically: publisher, user, and protocol — all transparent, all verifiable.

Protocol Architecture

Nine smart contracts deployed on Polkadot Hub (PolkaVM / pallet-revive):

  • PauseRegistry — Emergency circuit breaker
  • Timelock — Governance-delayed parameter changes
  • Publishers — Publisher registration, take rates, category bitmasks
  • Campaigns — Campaign lifecycle, budget escrow, open/targeted campaigns
  • GovernanceV2 — Conviction-weighted voting, anti-grief termination protection
  • GovernanceSlash — Symmetric slash mechanics for losing-side voters
  • Settlement — Batch claim verification, three-way revenue split
  • Relay — EIP-712 publisher co-signature attestation
  • ZKVerifier — Engagement proof verification (stub, Groth16 post-alpha)

Publisher Integration

Add the SDK to any page with two lines:

<script src="https://your-cdn/datum-sdk.js"
  data-categories="6,26"
  data-publisher="0xYOUR_PUBLISHER_ADDRESS"
  data-relay="https://your-relay.example.com"></script>
<div id="datum-ad-slot"></div>

data-categories declares which ad categories your site accepts (comma-separated IDs from the taxonomy below). data-publisher is your registered on-chain address. data-relay is the URL of your publisher relay bot (the extension uses this to request attestations and submit claims — no extension rebuild needed when the URL changes). The extension matches campaigns by category overlap, runs a Vickrey auction, and injects the winning ad into the slot.

Category Taxonomy

Campaigns and publishers declare categories using numeric IDs. Top-level categories (1–26) are used for on-chain bitmask matching. Subcategories provide finer-grained classification for client-side filtering.

Publisher Relay Bot

Publishers run a relay bot that co-signs user claim batches (EIP-712 attestation) and submits them on-chain via DatumRelay.settleClaimsFor(). The relay pays gas so users don't need to — the publisher is reimbursed from the campaign budget's take-rate share.

How It Works

  • User browses a publisher page with the DATUM SDK and extension installed
  • Impressions are tracked locally; claims accumulate in the extension
  • When the user signs a claim batch (EIP-712), the extension requests a publisher co-signature from the relay's /.well-known/datum-attest endpoint
  • The signed batch is sent to /relay/submit, where the relay verifies the user's EIP-712 signature, queues the batch, and submits it on-chain every 5 minutes
  • Settlement splits revenue: publisher (take rate), user (75% of remainder), protocol (25%)

Security

  • No private key exposure: Users sign claim batches locally using EIP-712 typed data. Only the signature is sent to the relay — private keys never leave the browser
  • Signature verification: The relay verifies every user signature before queuing. Forged or tampered batches are rejected
  • Rate limiting: Per-IP sliding-window limits (10 attestation/min, 5 submit/min)
  • Input validation: Address format, nonce range, batch size caps (max 100 claims)
  • HTTPS only: Exposed via Cloudflare tunnel; the extension rejects HTTP for non-local domains

Downtime Risk

This demo relay uses a Cloudflare Quick Tunnel, which assigns a temporary *.trycloudflare.com URL. The URL changes if the tunnel restarts. If the relay is offline (check the indicator above), claims will still accumulate in the extension — they can be submitted when the relay comes back online, or exported and submitted manually. No claims or earnings are lost due to relay downtime.

Run Your Own Relay

A relay bot template is included in the repository at docs/relay-bot-template/. To set up your own publisher relay: 

# 1. Register as a publisher on-chain (via the DATUM extension Publisher tab)
# 2. Copy the template
cp -r docs/relay-bot-template my-relay-bot
cd my-relay-bot

# 3. Install dependencies
npm install

# 4. Configure your publisher key and contract addresses
#    Edit .env (see .env.example in the template)
export PUBLISHER_KEY="0xYOUR_PRIVATE_KEY"
export RPC_URL="https://eth-rpc-testnet.polkadot.io/"

# 5. Start the bot
node relay-bot.mjs

# 6. (Optional) Expose via Cloudflare tunnel for remote access
cloudflared tunnel --url http://127.0.0.1:3400

See the template's README.md for full configuration options, systemd setup, and security hardening details.
